### Option A — User softens CLAUDE.md QQQ Outperformance Rule for diversifier role

**What**: User explicit-go to soften the OOS walk-forward window-mean rule for
candidates that satisfy:
- Full-period vs_qqq > 0 ✓
- Holdout (2025) vs_qqq > 0 ✓
- Per-window: full-period > 0 holdout > 0; window-mean rule waived for
  candidates pre-registered as `diversifier` role (not `core_alpha`).

**Effect on trial 9**: would qualify as DIVERSIFIER candidate (not core).
Track A acceptance β-stamp + evidence pack §4.6+§4.7 still required. Forward
init eligible.

**Risk**: changes project invariant scope. CLAUDE.md must be updated to
clarify the diversifier-role waiver. May open governance creep.

**Operator opinion**: this is a meaningful relaxation but with a clean
pre-registered scope (diversifier role only, not core). The window-mean rule
was originally designed for standalone core strategies. A diversifier in a
fleet legitimately can underperform QQQ in individual windows if the fleet's
combined NAV outperforms.

### Option B — D3b regime-aware mining objective

**What**: Replace IC_IR maximization with regime-conditional IC penalty:
penalize trials that have negative vs_qqq in BULL/RISK_ON regimes.

**Effect**: would steer mining toward composites that perform well in QQQ-favorable
regimes, reducing the validation-window vs_qqq dispersion.

**Cost**: ~1 week engineering work. New mining objective module + acceptance
gate updates + cycle yaml schema additions.

**Risk**: regime classification has its own uncertainty (CLAUDE.md QQQ Rule
table notes "QQQ is tech-concentrated; in pure bull markets even well-
diversified strategies may lag QQQ temporarily"). D3b may or may not solve the
problem; it might just shift the failure mode.

**Operator opinion**: worth doing eventually but not the highest-leverage next
move. The current 7-Tier-1 result shows the mining IS finding interesting
shapes; the issue is the gate, not the mining objective.

### Option C — Two-stage allocation architecture

**What** (per cycle04 collaborator advisory): separate asset allocation layer
(equity vs bonds vs commodities vs cash, per regime) from equity sleeve stock
selection. Trial 9 (and similar candidates) become DIVERSIFIER candidates
WITHIN the allocation framework, not standalone cores.

**Cost**: 4-6 weeks engineering. Asset allocation policy module + sleeve
isolation + capital routing + per-sleeve risk budgets + multi-sleeve aggregate
report. Substantial PRD.

**Effect on trial 9**: clean fit as bonds+cash+equity diversifier within the
allocation. Avoids forcing it to beat QQQ standalone.

**Risk**: large engineering investment. Cycle results so far (incl. trial 9)
don't FORCE this pivot — they motivate it but don't require it.

**Operator opinion**: directionally agree this is structurally cleaner long-
term. But 4-6 weeks of blind infrastructure investment without first validating
the candidate is risky. Recommendation: if user picks Option C, scope it as a
PRD with milestones; don't treat as auto-fire.

### Option D — Diversifier reframe (lightweight)

**What**: Reclassify trial 9 (and any similar future candidate) as DIVERSIFIER
within current single-stage framework. Lighter-touch than Option C: doesn't
build a full allocation layer, just allows manual `diversifier` role tagging
in candidate registry. Forward observation runs in parallel with RCMv1+Cand-2;
fleet allocator (Track B Step 6+) when unfrozen handles weighting.

**Cost**: 1-2 days engineering. `frozen_spec` schema extension (role tag) +
candidate registry update + Track A acceptance gate dispatch by role.

**Effect on trial 9**: enters fleet as diversifier (3rd candidate alongside
RCMv1+Cand-2), forward-observed for TD60. Track A acceptance β-stamp at
diversifier-role thresholds (lighter than core_alpha but not zero).

**Risk**: depends on Option A path. If Option A not approved, this option
auto-fails (CLAUDE.md QQQ Rule blocks even diversifiers from window-mean
underperformance per current invariant text).

**Operator opinion**: this is the LOWEST-COST path to actually exercising
trial 9's diagnostic value. Pairs with Option A. Together they unblock trial 9
without building 4-6 weeks of allocation infrastructure.

### Option E — Hold + observe; no further mining

**What**: Cycle #05 closes 0 nominee. RCMv1 + Cand-2 forward observation
continues. NO new mining cycles until a structural insight or trigger emerges.

**Effect on trial 9**: archived but not promoted; sits in `data/mining/rcm_archive.db`
with lineage track-c-cycle-2026-05-01-05.

**Risk**: project remains in "no working core under new framework" state
indefinitely.

**Operator opinion**: defensible but pessimistic. Trial 9 IS evidence of
mining productivity given relaxed gates; we shouldn't fully discard it.

### Option F — Universe expansion (D3a→D3b iteration on cross-asset side)

**What**: Add more cross-asset symbols (EEM/EFA/VNQ/DBC, possibly TIPS).
Re-run cycle #05's spec banning under the expanded universe.

**Cost**: ~1 week. Universe yaml update + factor coverage check + risk
cluster map extension + 200-trial mining + eval.

**Effect**: may produce additional Tier 1 candidates with stronger
diversification (raw NAV < 0.50 = `true_diversifier` band).

**Risk**: same structural attractor may persist. The 0.50 boundary may not
be breakable with US-equity-heavy long-only construction even with universe
expansion.

**Operator opinion**: complementary to Options A+D. Worth doing AFTER A+D
verified to work on trial 9. Not a substitute.
